# NMPA申报材料DOCX润色

## 1. 概述

本 skill 用于对 Word 文档中的申报资料文字进行专业化润色，使其更符合 NMPA 注册申报材料的常用表达习惯和正式文体要求。

润色重点包括：

- 术语规范化
- 表述正式化
- 句式和逻辑优化
- 前后一致性检查
- 申报方表达的专业度提升
- 在不改变原意的前提下增强材料的清晰度、完整性和可读性

本 skill 的定位是“申报方专业润色”，不是审评意见模拟、缺陷审查或合规审计。除非用户明确要求，不对材料进行过度审评化、质疑化或风险放大式改写。

---

## 2. 适用范围

适用于以下类型文档：

- 药品注册申报资料
- 临床试验申请资料
- 临床试验方案及修订说明
- 研究者手册
- 临床研究总结报告
- 非临床研究总结资料
- 药学研究资料
- 安全性更新资料
- 发补回复资料
- 沟通交流会议资料
- 医疗器械或体外诊断产品申报资料中的文字说明部分

---

## 3. 工作目标

润色后的文档应达到以下效果：

1. **表达更正式**
   - 避免口语化、宣传化和模糊化表述。
   - 使用注册申报资料中更常见的书面表达。

2. **术语更统一**
   - 统一药品名称、适应症、研究阶段、终点、统计集、安全性术语等。
   - 尽量采用 NMPA 及 ICH 语境下常用术语。

3. **逻辑更清楚**
   - 优化段落结构和句间衔接。
   - 使研究目的、方法、结果、结论之间的关系更清晰。

4. **语气更稳健**
   - 避免过度承诺或绝对化结论。
   - 对疗效、安全性、质量研究等结论采用客观、克制、专业的表达。

5. **格式尽量不变**
   - 尽可能保留原 Word 文档的版式、字体、编号、表格结构和图表标题。
   - 主要修改文字内容，不主动改变排版结构。

---

## 4. 工作流程

### 4.1 确认输入文件

用户提供 docx 文件路径后，确认文件存在且可读。

如用户未说明具体润色范围，默认润色以下内容：

- 正文段落
- 表格内文字
- 图题、表题
- 脚注中可编辑文字

默认不主动修改以下内容：

- 参考文献条目
- 法规条款编号
- 数据表中的数值
- 图片中的文字
- 已锁定域代码
- 自动目录和交叉引用字段

---

### 4.2 选择工作模式

如任务较大，可向用户确认工作模式：

- **全自动模式**：润色全部可编辑文本，完成后输出润色版文档和简要报告。
- **重点润色模式**：仅润色正文叙述性内容，尽量少动表格、标题和固定模板文字。
- **交互模式**：逐段展示“原文—润色后”，由用户确认后继续。

如用户未指定，默认采用 **全自动模式**。

---

### 4.3 备份原文件（同时作为工作文件）

将原始文件直接复制为 `{原文件名}_polished.docx`，此文件既作为备份也作为后续所有操作的唯一工作文件。

```bash
cp "{原文件}.docx" "{原文件}_polished.docx"
```

后续的解包、润色、打包均在此 `_polished.docx` 上直接进行，不再产生独立的备份版本。原始文件不被覆盖。

---

### 4.4 解析文档

使用 docx skill 的 unpack 流程从 `_polished.docx` 中提取 XML：

```bash
python scripts/office/unpack.py "{原文件}_polished.docx" {临时目录}/unpacked/
```

解析以下文件中的文本：

- `word/document.xml`
- 必要时解析 `word/header*.xml`
- 必要时解析 `word/footer*.xml`
- 必要时解析 `word/footnotes.xml`
- 必要时解析 `word/endnotes.xml`

---

### 4.5 提取文字段落

从 XML 中提取所有包含 `<w:t>` 文本内容的段落，并记录其 XML 位置，便于后续替换。

**最小粒度为段落。** 一个 `<w:p>` 内的所有 `<w:t>` 文本拼接为一个完整段落，作为润色的最小处理单元。不得将同一段落拆分为多个短句分别处理。

应处理：

- 正文段落
- 表格单元格内文字
- 图题和表题
- 项目符号或编号列表中的文字

应跳过或谨慎处理：

- 纯图片段落
- 自动目录
- 交叉引用字段
- 页码字段
- 参考文献列表
- 文献引用编号
- 公式
- 域代码
- 超链接 URL

**段落过滤规则**（跳过不需润色的段落）：

- 章节标题（以数字编号开头，如 "2.4.1"）
- 表题/图题（以"表"或"图"开头）
- 缩写定义（纯英文短文本）
- 核酸序列（含 `*` 修饰标记的碱基序列）
- 过短文本（中文少于 30 字符）
- 纯数字/参考文献编号行
- 不含中文字符的段落
- **文献/参考文献段落**：当检测到章节标题为"参考文献"或"References"时，从该标题开始至文档末尾的所有段落均跳过，不进行润色。若标题为"参考文献"但后续有非参考文献的正文附录，需人工判断处理。

---

### 4.6 润色文字

对每个提取的**完整段落**（非短句），使用当前环境 LLM 进行润色。润色规则参见 `references/nmpa_style.md`。

**关键约束：一个段落作为整体一次性润色，不得拆分为短句分别处理。**

润色应采用“申报方专业表达”原则，即在不改变原意、不新增事实、不放大或弱化结论的前提下，使文本更符合注册申报资料的正式表达习惯。

**核心原则：**

1. **保持原意**
   - 不改变原文事实、数据、研究设计、统计结果和结论方向。
   - 不添加原文未提供的新信息、新解释或新结论。

2. **表达规范**
   - 使用正式、客观、清晰的书面语。
   - 避免口语化、宣传化、情绪化和过度营销化表达。

3. **术语统一**
   - 统一全文中药品名称、疾病名称、研究阶段、终点名称、分析集名称、安全性术语和统计术语。
   - 对明显不一致的术语进行规范化替换。

4. **逻辑顺畅**
   - 优化长句和复杂句。
   - 增强句间衔接，使段落内部逻辑更加自然。
   - 在必要时调整语序，但不改变原文信息层级。

5. **结论稳健**
   - 保留“提示”“观察到”“可能”“尚需进一步验证”等限定性表达。
   - 避免将趋势性、探索性或描述性结果改写为确证性结论。
   - 避免使用“完全”“绝对”“显著优于”“安全性极佳”等过强措辞，除非原文已有充分支持。

6. **安全性表述准确**
   - 区分 AE、ADR、SAE、TEAE、AESI、DLT 等术语。
   - 不将“未发生严重不良事件”直接改写为“安全性良好”。
   - 安全性结论应保持客观、适度、与原文数据一致。

7. **不做过度审评化改写**
   - 不主动加入“需进一步验证”“解释需谨慎”“存在风险”等审评质疑性表达，除非原文本身已有类似限定或用户明确要求。
   - 不将普通申报资料润色为审评缺陷分析报告。
   - 如发现明显逻辑或术语问题，可在润色报告中温和提示，而非直接大幅改写正文。

---

### 4.7 替换原文

在 XML 中将原文替换为润色后文字。

**关键要求：**

- 只修改 `<w:t>` 标签内的文字内容。
- 不改动 `<w:rPr>`、`<w:pPr>` 等格式属性。
- 尽量保留原段落内加粗、斜体、上下标、颜色、字体和编号样式。
- 对一个段落中分散在多个 `<w:t>` 中的文字，应避免破坏 run 结构。
- 如无法安全替换复杂结构，应保留原文并在报告中说明。

---

### 4.8 打包输出

将修改后的 XML 等内容重新打包为 docx，覆盖写入同一工作文件（即 `_polished.docx`，覆盖解包前的版本）。

```bash
python scripts/office/pack.py {临时目录}/unpacked/ "{原文件}_polished.docx"
```

打包完成后，删除临时解包目录。

**重要：打包完成后，`_polished.docx` 即为最终润色版文档，不再额外生成一份独立输出。**

---

### 4.9 生成润色改动报告（docx版）

润色完成后，必须自动生成一份 docx 格式的改动报告，与原文档存放在同一目录。

#### 报告文件命名

```text
原文件: report.docx
报告: report_polish_report.docx
```

#### 报告生成逻辑（内置流程）

润色改动报告是整个工作流的内置环节，并非独立可选任务。其生成过程嵌入在第 4 节的各步骤中，具体如下：

**数据收集阶段**（第 4.5 步——提取段落时）：
- 在提取 body paragraphs 时，将每个段落的原始文本、`para_idx`、`t_texts_original` 保存到 `body_paras_meta.json` 中，作为报告的原文数据源。

**润色记录阶段**（第 4.6 步——润色时）：
- 每润色完一个段落，立即将 `{para_idx}\t{polished_text}` 记录到一个批处理文件（如 `batch_0_polished.txt`），格式如下：
  ```
  ===PARA 0===
  polished text...
  ===PARA 1===
  polished text...
  ```
- 所有批处理文件统一存放在同目录，作为报告润色后文本的数据源。

**报告生成阶段**（第 4.8 步——打包完成后执行）：
- 用 Python 脚本 `generate_report.py` 生成 Word 版改动报告，写入 `{原文件名}_polish_report.docx`。

脚本包含以下核心逻辑：

**1. 章节标题提取** —— 扫描 word/document.xml 中的所有 `<w:p>`，匹配 `^[\d\.]+\s` 正则模式提取章节编号和标题，构建 headings 列表：`[(para_idx, "2.4.1", "非临床策略概述"), ...]`。

**2. 段落归属匹配** —— 逐一遍历改动条目，用 `get_section_for_para(para_idx, headings)` 找到最接近的前驱章节：
```python
def get_section_for_para(para_idx, headings):
    best = ('', '前言')
    for h_idx, h_num, h_title in headings:
        if h_idx <= para_idx:
            best = (h_num, h_title)
        else:
            break
    return best
```

**3. 改动类型自动分类** —— 通过关键词模式匹配实现，不依赖 LLM，规则按优先级顺序如下：

| 匹配条件（原文或润色后含以下模式） | 分类标签 |
|---|---|
| `医学需求` / `临床需求` / `同一性` / `序列同源性` / `其它` / `其他` / `毒理` / `毒理学` / `临床实验` / `临床试验` / `病人` / `患者` / `受试者` / `供者` / `供体` / `双报批准` / `双报获批` / `临床批准` / `临床试验批准` | 术语规范 |
| `\b我们\b` / `申请人` / `翻译出的` / `翻译所得` / `所以` / `因此` / `故` | 语气正式化 |
| `剂量依赖性` / `~` / `—` | 句式优化 / 数字格式统一 |
| `，` / `。`（中英文标点替换） | 标点规范 |
| 以上都不匹配 | 句式优化（默认） |

多标签用 ` / ` 拼接。此规则集可在 skill references 中维护扩展。

**4. Word 报告生成** —— 使用 `python-docx` 库，按以下结构生成文档：

- **封面**：标题"申报资料润色改动报告"、文档名称、润色日期、润色模式、处理段落统计
- **第一章：修改统计摘要**
  - 统计总表（文档总段落数 / 提取段落数 / 实际润色数 / 未修改数 / 字符变化量）
  - 改动类型分布表（按类型统计涉及段落数）
- **第二章：典型修改示例**
  - 从每种改动类型中选 1 个代表段落
  - 加选 1 个最长改动段落（不超过 8 个示例）
  - 每个示例以 2×2 对比表格展示：改动前 | 改动后
  - 超过 300 字自动截断并标注"仅展示前300字"
- **第三章：逐段改动明细**
  - 五列表格：序号 | 所属章节 | 改动类型 | 改动前（原文） | 改动后（润色后）
  - 超过 200 字自动截断并标注全文长度
  - 列宽按内容比例设置

如环境未安装 `python-docx`，先执行 `pip install python-docx`。

## 5. 润色输出

润色完成后，**必须**输出以下文档：

1. **润色版文档**：`{原文件名}_polished.docx`（原始文件的备份 + 润色后的最终版本）
2. **润色改动报告**：`{原文件名}_polish_report.docx`（改动对照明细）

不再产生单独的备份版本。`_polished.docx` 既是工作文件也是最终输出。

## 6. 交互模式细节

### 6.1 最小粒度

**一个段落 = 一次润色操作。** 不得将段落拆分为短句逐句处理。一个 `<w:p>` 内的所有文字拼接后作为整体润色、整体展示、整体确认。

### 6.2 展示方式

由于段落可能较长（200-500+ 字符），必须确保原文和润色后文字完整展示，**不得截断**。具体做法：

1. **在消息正文中**输出完整原文和润色后文字（使用 markdown 引用块 `>` 包裹），并列出主要修改点。
2. **在 AskUserQuestion 中**仅放置简短的选项按钮（接受/保留原文/手动编辑），**不重复放置段落全文**。

示例格式：

```
---
**段落 [3/119]** — `para_idx=306` (45字)

**原文：**
> mRNA核酸序列（CDS和polyA）经过优化，但翻译出的Cas9核酸酶蛋白序列完全一致。

**润色后：**
> mRNA核酸序列（编码区和polyA尾）经优化后，翻译所得Cas9核酸酶蛋白序列保持一致。

**主要修改：**
1. "CDS" → "编码区"（首次出现使用中文全称）
2. "但翻译出的" → "翻译所得"（去掉不当转折）
---
```

然后用 AskUserQuestion 仅询问选择。

### 6.3 用户操作

每段等待用户选择：
- **接受**：采用润色后文字替换原文
- **保留原文**：跳过此段，不做修改
- **手动编辑**：由用户提供最终文字版本

如用户连续接受多个相似修改，可询问是否对同类问题批量处理。

## 8. 依赖

- docx skill：用于 unpack / pack docx 文件
- 当前环境 LLM：用于文本润色
- Python XML 处理工具：用于解析和替换 Word XML（xml.etree.ElementTree）
- `python-docx`：用于生成 Word 格式改动报告（`pip install python-docx`）
- pandoc：可选，用于辅助文本提取或对照检查

---

## 9. 质量控制

润色完成后应进行基础检查：

1. 文档能否正常打开。
2. 总页数、章节结构、表格结构是否基本保持。
3. 是否存在明显乱码、空段落异常或 run 合并错误。
4. 数字、单位、p 值、百分比、样本量等是否未被意外修改。
5. 章节编号、表图编号、参考文献编号是否未被破坏。
6. 润色报告是否列明主要修改类型和需确认问题。

如发现异常，应回退到原始文件，重新复制为 `_polished.docx` 后重新处理。
